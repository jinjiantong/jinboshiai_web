# 飞书多维表格数据变更检测方案

## 📋 方案概述

本文档描述了如何利用飞书多维表格 API 提供的时间戳字段实现高效的数据变更检测机制，避免不必要的数据传输。

## 🔍 飞书 API 支持的字段

飞书多维表格 API 在返回记录时会包含以下关键时间戳字段：

| 字段名 | 类型 | 说明 |
|:---:|:---:|:---|
| `created_time` | number | 记录创建时间（毫秒时间戳） |
| `last_modified_time` | number | 记录最后修改时间（毫秒时间戳） |

### API 响应示例

```json
{
  "record_id": "rec0qHGvmiGoIF",
  "created_time": 1780153017000,
  "last_modified_time": 1780316143000,
  "fields": {
    "老师姓名": "张老师",
    "联系电话": "13800138000"
  }
}
```

## 🛠️ 实施方案

### 方案一：服务端缓存 + 时间戳对比（已实现）

在服务端维护缓存，记录每个记录的 `last_modified_time`，客户端请求时带上本地数据的最后修改时间，服务端对比判断是否需要返回新数据。

#### 实现逻辑

```typescript
// 1. 服务端缓存结构
const teachersCache: Map<string, {
  record: any;
  last_modified_time: number;
}> = new Map();

// 2. 客户端请求时带上 last_modified 参数
GET /api/teachers?last_modified=1780316143000

// 3. 服务端对比判断
if (服务端缓存的 last_modified_time <= 客户端 last_modified) {
  return 304 Not Modified
}

// 4. 否则返回新数据
return 200 OK { records: [...], last_modified_time: 最新时间戳 }
```

#### 优势

- ✅ 减少网络传输（数据未变化时只返回状态码）
- ✅ 降低飞书 API 调用次数
- ✅ 用户体验更好（秒级响应）

#### 劣势

- ⚠️ 需要维护服务端缓存
- ⚠️ 缓存过期策略需要仔细设计
- ⚠️ 不适合分布式部署（需要 Redis 等共享缓存）

---

### 方案二：客户端本地缓存 + 条件请求（推荐）

客户端本地存储数据及其 `last_modified_time`，请求时通过自定义请求头或参数传递，服务端判断后决定返回完整数据还是 304。

#### 实现逻辑

```typescript
// 前端代码
async function fetchTeachers() {
  const localData = localStorage.getItem('teachers')
  const localTimestamp = localStorage.getItem('teachers_timestamp')
  
  const response = await fetch('/api/teachers', {
    headers: {
      'X-Data-Last-Modified': localTimestamp || ''
    }
  })
  
  if (response.status === 304) {
    console.log('数据未变化，使用本地缓存')
    return JSON.parse(localData)
  }
  
  const data = await response.json()
  const newTimestamp = response.headers.get('X-Data-Last-Modified')
  
  // 更新本地缓存
  localStorage.setItem('teachers', JSON.stringify(data))
  localStorage.setItem('teachers_timestamp', newTimestamp)
  
  return data
}
```

```typescript
// 服务端代码
export async function GET(request: Request) {
  const clientTimestamp = request.headers.get('X-Data-Last-Modified')
  
  if (clientTimestamp) {
    // 对比服务端数据时间戳
    if (!hasDataChanged(clientTimestamp)) {
      return new Response(null, { status: 304 })
    }
  }
  
  // 返回数据，并带上新的时间戳
  return new Response(JSON.stringify(data), {
    headers: {
      'X-Data-Last-Modified': latestTimestamp
    }
  })
}
```

#### 优势

- ✅ 无需服务端维护缓存
- ✅ 适合分布式部署
- ✅ 响应更快（完全由客户端控制）

#### 劣势

- ⚠️ 需要前端额外处理逻辑
- ⚠️ 本地存储有容量限制
- ⚠️ 需要处理存储同步问题

### 方案三：WebSocket 实时推送（进阶方案）

通过飞书 Webhook 订阅数据变更事件，实时推送更新到客户端。

#### 实现逻辑

```typescript
// 1. 配置飞书 Webhook
POST /webhook/subscribe
{
  "table_id": "tblxxx",
  "events": ["record.create", "record.update", "record.delete"]
}

// 2. 接收变更事件
POST /webhook/callback
{
  "event": "record.update",
  "table_id": "tblxxx",
  "record_id": "recxxx",
  "data": { ... }
}
```

#### 优势

- ✅ 实时性强
- ✅ 体验最佳
- ✅ 节省轮询资源

#### 劣势

- ⚠️ 实现复杂度高
- ⚠️ 需要服务端保持连接
- ⚠️ 需要处理重连和消息丢失

## 🎯 我们的实现

当前系统采用了**方案一（服务端缓存 + 时间戳对比）**，具体实现：

### 服务端缓存策略

```typescript
// 单条记录缓存（30秒TTL）
interface CachedTeacher {
  record: any;
  last_modified_time: number;
}

// 列表缓存（30秒TTL）
const teachersListCache = {
  records: any[];
  last_fetch_time: number;
}
```

### 缓存失效策略

- **主动失效**：增删改操作时清除相关缓存
- **TTL过期**：30秒后自动过期，重新获取
- **版本对比**：基于 `last_modified_time` 判断是否变化

### API 响应

```typescript
// GET /api/teachers
{
  "code": 0,
  "data": [...],
  "cached": false,           // 是否来自缓存
  "last_modified_time": 1780316143000  // 最新修改时间
}
```

## 🔧 优化建议

### 短期优化

1. **增加本地缓存**：在浏览器 localStorage 中缓存数据
2. **智能刷新**：用户操作（添加/编辑/删除）后主动刷新
3. **增量更新**：只获取变化的部分，而不是全部

### 长期优化

1. **引入 Redis**：使用 Redis 共享缓存支持多实例部署
2. **WebSocket 推送**：实时同步数据变更
3. **CDN 加速**：静态资源缓存到边缘节点
